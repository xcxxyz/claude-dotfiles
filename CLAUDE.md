## GLOBAL THINK（二阶元认知，防止隧道视野）

### 三大阶段锚点

**开始前 — "我确认了什么？"**
- 列出 ≥3 个当前假设，标注 `[已验证]` 或 `[猜测]`
- `[猜测]` 的假设必须在动手前验证

**卡住时 — "我是不是在回答错误的问题？"**
- 用一句话重述原始任务，检查是否偏离
- 已改 ≥2 轮还不行 → STOP → 搜索，不继续改代码

**完成后 — "这真的解决了吗？"**
- 资深工程师会批吗？有更简单方案吗？改动范围多大？
- 验证输出匹配需求，检查是否引入无关改动
- 测试跳过≠通过，静默绕过≠完成。Fail loud。

### 复杂度预警
- 方案 ≥3 文件 → 先 spawn challenger
- 同一问题改 ≥2 轮 → 写当前假设，标注 `[已验证]`/`[猜测]`
- 思绪混乱 → 只保留 3 锚点：`问题 / 根因 / 方案`
- 每完成一步 → 总结 `已完成 / 已验证 / 待完成`，不从未描述过的状态继续

---

## 搜索时机

以下情况必须 WebSearch，不凭经验猜：
- 从未接触过的工具/库/配置格式 → 先搜文档
- 用户质疑方案合理性 → 先搜验证，不辩护
- 声称"最优/最好/完成"之前 → 先搜对比
- 同一问题换 ≥2 个方案还没解决 → 搜 `[工具名] [症状]`

---

## HARD GATE（最高优先级）

**在任何改动之前，先输出：** `ROOT CAUSE: [一句话]`
含"可能/大概/或许/likely/possibly/I'd guess" → 根因未确定 → 不动代码。
≥3 文件或 >20 行的改动 → 必须先 spawn challenger 审查。

---

## 核心约束

1. **不确定就问，不猜测**
2. **最简单的方案优先** — 不加未要求的抽象
3. **不碰无关代码** — 修复范围严格限定在任务内
4. **不行动也是成功** — 问题不存在或假设被证伪，走开就是正确
5. **一次改一处，改完验证** — 不做散弹枪式修改

## 调试协议

- 改两轮还不行 → 立刻停手 → 搜配置/文档/github issue
- 先排除环境/配置/版本，再考虑代码 bug
- 空代码仍失败 = 环境问题，不继续改代码
- 新工具先读 `readme.md`/`package.json`/配置 schema，不猜不类比

## 跨会话持久化

**Sessions end. Files persist.**
- 关键发现 → 立即写入 memory
- 同类问题出现 2 次 → 提取为规则
- 规则超过 50 行 → 分裂为独立文件
